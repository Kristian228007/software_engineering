# Оптимизация производительности

## 1. Кеширование (Redis)

### Какие данные кешируем
- `GET /api/v1/recipes` → ключ `recipes:all`, TTL 60 секунд
- `GET /api/v1/recipes/{id}/ingredients` → ключ `recipe:{id}:ingredients`, TTL 60 секунд

### Стратегия
- Cache-Aside (Lazy Loading)
- Инвалидация при `POST /api/v1/recipes` и `POST /api/v1/recipes/{id}/ingredients`

### Реализация
- `CacheService` — обёртка над Poco::Redis::Client
- `JsonUtils` — сериализация рецептов/ингредиентов в JSON

## 2. Rate Limiting

### Алгоритм
- Fixed Window Counter (Redis INCR + EXPIRE)

### Лимиты
- 100 запросов в минуту для `GET /api/v1/recipes` и `GET /api/v1/recipes/{id}/ingredients`

### Заголовки
- `X-RateLimit-Limit: 100`
- `X-RateLimit-Remaining`
- `X-RateLimit-Reset: 60`

### HTTP 429
Возвращается при превышении лимита.

## 3. Анализ производительности

### Как кеширование улучшает производительность
- **До кеша**: `GET /recipes` выполняет сложный `LEFT JOIN` + `json_agg` в PostgreSQL (~50-80 мс)
- **После кеша**: данные из Redis (~2-5 мс), ускорение в **10-20 раз**
- Снижение нагрузки на БД: читающие запросы не попадают в PostgreSQL

### Как rate limiting улучшает производительность
- Не даёт злоумышленникам или багам отправлять слишком много запросов
- Не даёт "положить" базу данных из-за перегрузки
- Все пользователи получают одинаково быстрый ответ

### Метрики для мониторинга

| Метрика            | Как измерить                           |
| ------------------ | -------------------------------------- |
| Cache hit rate     | `hits / (hits + misses)`               |
| Время ответа (p95) | Замерить до и после кеша               |
| Кол-во 429 ответов | Счетчик в RateLimiter                  |
| Нагрузка на БД     | Количество запросов в секунду до/после |
